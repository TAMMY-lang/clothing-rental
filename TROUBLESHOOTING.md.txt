# 上线后故障排查清单

这份文档用于上线后遇到问题时快速排查。先找到你遇到的问题，再按步骤检查。

## 一、先判断是哪一层出问题

```text
网页打不开 → 多半是前端 Vercel 问题

网页能打开，但登录/商品/订单失败 → 多半是后端 Railway 或环境变量问题

后端 health 失败 → 后端服务启动失败

后端能打开，但数据为空或报错 → 多半是数据库连接或初始化问题


Plain Text


快速检查地址：

| 检查项 | 打开方式 |
|---|---|
| 前端 | 打开 Vercel 网址 |
| 后端健康检查 | 打开 `https://你 的后端网址/health` |
| 数据库 | 打开 Supabase 项目 |
| 后端日志 | Railway → Deploy Logs |
| 前端日志 | Vercel → Deployment Logs |

## 二、服务启动失败

### 常见表现

- Railway 部署失败。
- `/health` 打不开。
- 后端日志出现红色错误。
- 前端提示接口无法连接。

### 先看日志

进入 Railway：

```text
项目 → 后端服务 → Deploy Logs


Plain Text


### 常见原因 1：环境变量缺失

检查 Railway 的 `Variables` 是否有：

```text
NODE_ENV=production PORT=3000 DATABASE_URL=... CORS_ORIGIN=... JWT_SECRET=...


Plain Text


如果少了 `JWT_SECRET`，登录功能会异常。

如果少了 `DATABASE_URL`，后端无法连接数据库。

### 常见原因 2：Root Directory 设置错了

Railway 后端 Root Directory 必须是：

```text
backend


Plain Text


如果写成根目录，Railway 找不到正确的 `package.json`。

### 常见原因 3：构建命令错了

后端 Build Command 建议：

```bash
npm ci && npx prisma generate && npm run build


Plain Text


Start Command：

```bash
npm start


Plain Text


## 三、数据库连接失败

### 常见表现

日志里出现：

```text
Prisma Can't reach database server Authentication failed DATABASE_URL


Plain Text


### 检查 1：Supabase 是否正常

打开 Supabase，确认项目没有暂停。

如果是免费项目，长时间不用可能会暂停，需要手动恢复。

### 检查 2：DATABASE_URL 是否正确

正确格式类似：

```text
postgresql://postgres:你的密码@db.xxxxx.supabase.co:5432/postgres?schema=public


Plain Text


注意：

- 密码不能写错。
- 如果密码里有特殊符号，建议在 Supabase 里重新设置一个简单但安全的密码。
- 最后建议带上 `?schema=public`。

### 检查 3：是否初始化数据库

在 Railway Shell 执行：

```bash
npx prisma db push npm run seed


Plain Text


如果没执行，页面可能没有商品和基础账号。

## 四、图片上传失败

### 常见表现

- 上传图片时报错。
- 商品发布后图片不显示。
- 刷新后图片丢失。

### 检查 1：图片格式

支持：

```text
jpg png webp


Plain Text


不支持的格式请先转换。

### 检查 2：图片大小

单张图片不要超过：

```text
5MB


Plain Text


如果超过，先压缩图片。

### 检查 3：是否登录商家账号

上传图片需要商家权限。

如果登录的是普通用户，上传会失败。

### 检查 4：Railway 文件存储问题

Railway 本地文件系统不适合作为长期图片存储。

测试时可以用，正式经营建议接入对象存储：

```text
阿里云 OSS 腾讯云 COS Cloudflare R2 Amazon S3


Plain Text


如果你发现“刚上传能显示，过一段时间不显示”，大概率是平台文件系统被重置。

## 五、支付失败

### 当前支付状态

当前系统是模拟微信支付流程。

也就是说：

```text
可以测试下单和支付流程 不能直接真实收款


Plain Text


### 常见表现

- 点击支付失败。
- 提示需要支付密码。
- 支付密码错误。

### 检查 1：是否设置支付密码

用户支付前需要设置 6 位数字支付密码。

如果提示：

```text
请先设置支付密码


Plain Text


需要先调用设置支付密码，或后续在页面中补充设置入口。

### 检查 2：支付密码是否输错

错误 3 次会锁定：

```text
30 分钟


Plain Text


等待后再试。

### 检查 3：真实收款还没接入

如果你要真实收款，需要准备：

- 微信商户号。
- 微信支付 AppID。
- 商户证书。
- 支付回调地址。

当前文档里的支付是测试模式。

## 六、登录失败

### 常见表现

- 提示密码错误。
- 提示需要图形验证码。
- 提示账号锁定。
- 提示陌生设备。
- 提示 TOTP。
- 提示账号已在其他设备登录。

### 情况 1：密码错误

确认是否使用正确账号：

```text
商家：13900000000 管理员：13700000000 普通用户：13800000000


Plain Text


上线后如果你改过密码，以新密码为准。

### 情况 2：需要图形验证码

连续输错 3 次后，需要图形验证码。

测试验证码：

```text
1234


Plain Text


### 情况 3：账号锁定

连续输错 5 次：

```text
锁定 15 分钟


Plain Text


连续输错 10 次：

```text
锁定 1 小时，需要联系客服或管理员处理


Plain Text


### 情况 4：陌生设备登录

系统检测到新设备时，会要求短信验证码。

测试验证码：

```text
123456


Plain Text


### 情况 5：账号已在其他设备登录

系统开启单设备登录。

如果你在新电脑登录，旧电脑会被下线。

这是正常安全保护。

## 七、2FA 无法验证

2FA 指 TOTP 动态验证码。

### 常见表现

- 输入 6 位码后提示错误。
- 每次都验证失败。
- 换手机后找不到验证码。

### 检查 1：手机时间是否准确

TOTP 依赖时间。

请打开手机：

```text
设置 → 日期与时间 → 自动设置


Plain Text


打开自动时间后再试。

### 检查 2：是否输入了最新验证码

动态码 30 秒更新一次。

如果当前验证码快结束了，等下一组再输入。

### 检查 3：是否扫错账号

如果身份验证器里有多个账号，确认选择的是当前平台的账号。

### 检查 4：手机丢失或 App 删除

如果商家账号无法登录，需要：

1. 使用管理员账号登录。
2. 查看用户。
3. 对商家账号重置安全设置。

当前后台已支持管理员查看用户和重置密码。TOTP 强制解绑可后续做成按钮；紧急情况可由技术人员在数据库里把：

```text
totpEnabled=false totpSecret=null


Plain Text


## 八、前端打开正常但数据加载失败

### 检查 Vercel 环境变量

进入 Vercel：

```text
项目 → Settings → Environment Variables


Plain Text


确认：

```text
VITE_API_BASE_URL=https://你 的后端网址


Plain Text


不要填：

```text
http://localhost:3000


Plain Text


因为线上用户访问不到你的本地电脑。

### 修改后要重新部署

Vercel 修改环境变量后，要点击：

```text
Redeploy


Plain Text


## 九、跨域错误

### 常见表现

浏览器控制台出现：

```text
CORS blocked by CORS policy


Plain Text


### 解决方法

进入 Railway 后端环境变量：

```text
CORS_ORIGIN=https://你 的前端网址


Plain Text


例如：

```text
CORS_ORIGIN=https://clothing-rental.vercel.app


Plain Text


保存后 Railway 会自动重启。

## 十、订单状态不对

### 用户说已经付款，但商家看不到

检查：

- 用户是否真的完成支付。
- 支付是否只是打开了页面，没有确认。
- 后端日志是否有支付回调。

当前是模拟支付，测试时会自动确认支付。

### 用户归还后没有到账

检查订单是否完成验收。

资金到账规则：

```text
验收后进入 SETTLING 达到结算日期后变成 SETTLED


Plain Text


如果要手动跑结算，商家后台有结算触发接口。

## 十一、资金和提现问题

### 提现按钮点不了或失败

检查：

- 可用余额是否大于 100 元。
- 是否绑定提现方式。
- 是否还有提现申请在审核。

### 为什么押金没进入商家余额

这是正确的。

规则是：

```text
商家入账 = 租金 + 快递费 + 试穿费 + 清洁费 押金退给用户


Plain Text


押金退款会记录为：

```text
DEPOSIT_REFUND


Plain Text


## 十二、紧急处理建议

如果线上出现严重问题：

1. 先不要连续乱改设置。
2. 截图错误页面。
3. 复制 Railway 或 Vercel 的错误日志。
4. 记录出问题的账号、订单号、时间。
5. 如果影响用户下单，先在店铺公告说明。

记录模板：

```text
问题时间： 用户手机号： 订单号： 操作步骤： 错误提示： 截图： 日志：


Plain Text
